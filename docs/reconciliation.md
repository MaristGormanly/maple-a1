# Design Doc Reconciliation

## Delivered vs. Specified

The design doc described MAPLE A1 as an instructor-driven grading assistant that combines deterministic test execution with AI-assisted rubric feedback. The implementation delivered the core MVP version of that plan: an Angular/FastAPI application with authenticated instructor workflows, GitHub repository submission through `POST /api/v1/code-eval/evaluate`, instructor PAT-based repository access, preprocessing, SHA/rubric-digest caching, asynchronous pipeline execution, and status polling.

On the backend, the project now persists users, assignments, rubrics, submissions, evaluation results, and style-guide chunks. The evaluation pipeline runs student code in Docker sandboxes with language profiles, resource limits, read-only mounts, log normalization, structured test parsing, and deterministic scoring. The AI layer also follows the planned shape: redaction before external calls, a centralized LLM wrapper with retries and fallback, Pass 1/Pass 2/Pass 3 orchestration, schema validation and repair, AST/code chunking, linter hooks, and pgvector-backed style-guide retrieval. On the frontend, the delivered status and review UI shows pipeline progress, test summaries, criteria scores, style findings, diff recommendations, and instructor approve/override controls.

## What Changed and Why

Several commitments were modified or narrowed. Student self-service remained outside the MVP. Instructors submit repositories on students' behalf, which reduced authentication and credential-management complexity while matching the pilot workflow. The design also assumed each assignment would use a separate instructor test-suite repository. The implementation added an `auto_discover` mode so assignments with tests already in the submitted repository can still be evaluated. That makes the system more flexible, but it means the test source is not always exactly the one described in the original plan.

AST-aware chunking was implemented most strongly for Python. JavaScript, TypeScript, and Java use a conservative regex fallback instead of full language parsers. This was a practical scope decision: it preserved Pass 2 style review without adding fragile parser dependencies late in the project. The RAG pipeline and style-guide storage exist, but their quality depends on successful ingestion, embeddings, and configured API keys. Similarly, the formal DigitalOcean pilot and 10-student evaluation metrics remain deployment and evaluation deliverables, not proven implementation facts.

## What We Learned

Spec driven development has been a great step forward for AI assisted software development. However, there are still many things that need to be addressed in order to ensure that software can be created reliably. Firstly, we learned that context is all. We realized that as the project began growing, it became more and more difficult to manage requirements even though they were specified in the requirements document. The complexity of the system made it more and more difficult for even Opus 4.7 with a one million context window to get an understanding of the components of the codebase. This wasn't necessarily the [design-doc.md](design-doc.md)'s fault, it was a development misunderstanding. We mis-interpreted the scope and the scale that maple-a1 was going to be. For a smaller side project, it is much easier to have a single design doc, have an agent reference it, make some modifications along the way, and have a pretty decent project. However, maple-a1 was an ambitious task that easily got lost in the context bloat and mess that is still a challenge for todays agents.

Although we came to this realization too deep within the project, we realized for future projects we should go back to the drawing board. We thought to ourselves, how would we go back and make the process more determinsitic. Our solution was to lean on the tools we used everyday even before agentic coding became main stream. We realized that markdown files can be too maleable and context can get lost within them. We realized that if we designed a spec, and created milestones, tasks, and subtasks for the spec, the context window can be tamed. We then wrote up a skill that instructed an agent to manage the github remote repository issues. When the tasks were moved to a remote location in a determinstic tool like github, the agent would have a better time getting the context for implementing the one milestone, task, or subtask it is focusing on. We also added a note to add a markdown file that highlights the interfaces each milestone is exposing. When an interface is confirmed to be working as intended for a milestone, the abstraction layer prevents the agent from becoming distracted from what really matters. When the agent is working on future modules and milestones that need to interface with a previous component, the agent can confidentally reference the previous modules it implemented without risking confusion from excess context.

We also learned that test first development is key. Although test file creation can also be hallucinated, looking through the tests briefly and then letting the agent write code that satisfies the tests will 9/10 figure out the issue based on the test. The relationship becomes the test checking the code and the code checking the tests. This relationship makes the development process of any task more reliable.

In creating these skills, we realized that having them documented is really important. That is why we created a personally shared repository that we will carry and share with all of our projects to ground them in the scope and context that matters. Github is for saving tasks, tests are for validating output with two-way relationship, and markdown files are for documenting integrations and identifying interfaces for general abstraction.
